# AI Architecture Notes

The canonical architecture document is `ARCHITECTURE.md`. This file is its compact
executable-context mirror.

It does not own capability maturity. The bullets below state what each contract is and
what it connects to; none of them state whether a gate is Contract Verified, Integrated,
or Operational. Read maturity from `PROJECT_STATE.md` and the evidence behind it from
`docs/verification-log.md`.

A contract description here MUST state what the contract connects to, not only what it
is. A described-but-unconnected contract is how the `completion` conflict happened:
this file described each queue, runtime, and result contract correctly while omitting
the ordered connection backlog between them, so the missing middle transition was not
visible. `ARCHITECTURE.md` holds that backlog under Gate 8 Connection Sequence And
Completion Boundary, and Completion Semantics states the three distinct facts that
`completion` names.

Current direction:

- Enhancer is a self-hosting AI Development Operating System.
- Delivery Gate 0 Context, planning, Assisted Loop, repeated-loop safety, ToolResult, VerificationEvidence, and governance contracts are connected through an authority-preserving lifecycle test; the bounded read-only Tool, persisted evidence, Tool-driven Agent Loop, verification, and RunRecord boundaries compose into the first supported read-only CLI.
- Executable repository context loads the seven governed `.ai/` documents before the eight canonical root documents.
- The Planner follows the canonical Delivery Gate / Specified - Next grammar and is regression-tested against the actual Enhancer Roadmap.
- Capability maturity is Specified, Contract Verified, Integrated, Operational, or Released; do not use Implemented alone as a roadmap state.
- A cross-cutting Product Journey and Evaluation Track complements, but never replaces, Delivery Gates. Four initial journeys cover governed bug repair, bounded feature delivery, evidence-backed codebase explanation, and interrupted-run recovery; release evidence uses versioned fixtures, predeclared thresholds, explicit denominators, held-out regression checks, and immutable run/model/policy/revision/evaluator provenance.
- The evaluation harness measures task success, incorrect changes, retry/recovery, cost/time, user intervention, post-verification regression, and multi-agent delta. Multi-agent comparisons use the same fixture revision and comparable budget envelope as the single-agent baseline; confidence or cherry-picked demonstrations are not evidence.
- Delivery Gate 1 integrates ToolRequest, Tool, ExecutionPolicy, ToolExecutor, and one allowlisted read-only filesystem Tool.
- Delivery Gate 2 integrates atomic complete-evidence persistence, resolvable references, and integrity validation.
- Delivery Gate 3 connects approved work and real Tool results to the bounded Agent Loop and stops success at `AWAITING_VERIFICATION`.
- Gate 3 hardening derives structured task approval from active repository context, uses typed Tool failure codes and semantic evidence digests, and restricts state construction.
- Delivery Gate 4 integrates sequential independent read-file verification, verified-only completion, controller-bound execution policy, and atomic lifecycle-valid replayable RunRecords.
- Pre-operational hardening isolates each Tool invocation, constrains representable timeouts, integrity-protects complete Evidence and RunRecord envelopes, strictly preserves UTF-8, bounds and contains startup context, and removes the Gradle 9 JUnit runtime deprecation.
- Evidence uses a truthful `EvidenceStoragePolicy` for the enforced per-artifact content bound; no time-based retention, expiry, or cleanup is claimed until separately authorized and designed.
- Delivery Gate 5 provides the supported `EnhancerCli` `run` and `replay` commands with bounded output, stable exit codes, verified-only completion, and durable replay.
- Gate 0 integration proves planning -> explicit external activation -> verified Gate 5 execution and replay without automatic Proposal approval or a second production orchestrator.
- Delivery Gate 6 Workspace and Project Brain Foundation covers the snapshot, view, graph, query, projector, and collector sub-capabilities, and the production CLI `run` path composes the view and graph. Diagnostics, terminal-session, and active/selected-file observation moved to Gate 12 by the 2026-07-15 user-approved re-scope decision. The snapshot identity is not stored in the RunRecord; Gate 7 envelopes own cross-handoff identity.
- Delivery Gate 7 Event Bus and IPC Foundation covers a reference-only `MessageEnvelope`, a deterministic single-threaded `InProcessMessageBus`, and a transport-neutral `MessageTransport` boundary spanning bounded payloads, topic fan-out, single-consumer queues, idempotency, journal replay, failure isolation, bounded retry and dead-letter re-delivery, correlation cancellation, run-to-completion ordering, finite non-blocking backpressure, and hop-level transport acceptance without granting authority. Persistence, threading, production wiring, and concrete adapters remain deferred. The current `Specified - Next` gate is stated in `ROADMAP.md` and `PROJECT_STATE.md`, not here.
- Gate 7's connected work-message queue path: `WorkMessagePublisher` derives an envelope from a matching real approved task and Workspace snapshot, the in-process bus delivers/journals/replays it, and `WorkItemAdmissionHandler` retains it unchanged in Gate 8 admission without duplicate work. Result/control/handoff and causation flows, topic and reliability branches, and `MessageTransport` have no named real production connection.
- Gate 8 admission: immutable `WorkItem` admission retains one unchanged Gate 7 work envelope, adds a distinct canonical work identity and bounded required capability, and exposes approved task, snapshot, run, and Tool scope only as projections. It adds no runtime state, persistence, lease, or authority; the dependency-ready single-worker Scheduler queue is its first consumer.
- Gate 8's dependency-ready single-worker queue: immutable queued work carries up to 256 earlier-admitted dependency identities, a run-scoped queue admits at most 4096 work items, FIFO selection releases only fully completed dependencies, and exactly one active slot exists. The base queue remains in-memory and creates no lease, failure policy, worker execution, or authority; the separate durable wrapper supplies schema-v1 state and restart recovery.
- Gate 8 durable queue state: one canonical queue identity and logical run persist through a bounded schema-v1 integrity-checked atomic snapshot, every enqueue/claim/completion persists before exposure, and restart recovery requeues interrupted active work in admission order under explicit at-least-once semantics. The adapter preserves exact WorkItem/envelope provenance and fails closed on missing, corrupt, oversized, structurally invalid, or unsupported state. It has no lease, fence, worker, external-effect deduplication, migration beyond v1, snapshot history, or power-loss claim for parent-directory metadata.
- Gate 8 durable Goal/AgentRun lifecycle state: one Goal retains one exact WorkItem, one schema-v1 AgentRun advances only through Planning, Ready, leased Executing, Awaiting Verification, and one typed terminal result, and every revision is persisted before exposure. A matching ResultPayload with Verified status alone completes the Goal; other verification states fail explicitly.
- Gate 8 fenced AgentRun ownership: `READY` acquisition issues a bounded single-owner lease with a persisted monotonic fence, injected time, and exclusive expiry; renew and execution completion require the current unexpired owner/fence, while restart or explicit reclaim at expiry durably returns execution to `READY` and makes the former owner stale. It grants no Tool authority and adds no worker, effect fencing, retry, multi-process lock, or clock-skew protocol.
- Gate 8's durable queue-to-lifecycle dispatch path: `DurableAgentRunDispatcher` uses an existing active WorkItem or durably claims the next ready item, creates or recovers the exact matching Goal, resumes the persisted AgentRun prefix, and returns its current fenced lease. Queue and runtime artifacts remain separate; partial runtime failure leaves a recoverable active claim, repeated same-owner calls are idempotent, and no Tool, external effect, queue completion, or cross-store transaction is added.
- Gate 8 execution acknowledgement is not queue completion: fence-checked `EXECUTING -> AWAITING_VERIFICATION` releases the lease only. The Scheduler queue records a terminal `WorkItemDisposition`: `completeActiveVerified` enters the dependency-satisfaction set `completedWorkItemIds`, while `failActive` enters a separate `failedWorkItemIds` set that never satisfies dependents, so failed work leaves dependents blocked with the cause held in the runtime/RunRecord. The queue stays active through verification and each terminal disposition then releases the slot; the schema-v1 partition is `pending + active + verified + failed`, the disposition persists in place with exact restart recovery, and pre-existing local snapshots fail closed.
- Gate 8 RunRecord-backed result-path finalization: `DurableAgentRunFinalizer` connects a resolved RunRecord to the runtime terminal state and the matching queue disposition in one recoverable, idempotent order (resolve RunRecord -> runtime `recordResult` -> queue disposition), deriving the disposition from the runtime terminal status so the two stores cannot diverge. It resolves (never persists) the RunRecord, binds it to the Goal on taskId plus sourceDocument, fails closed on a missing/corrupt record leaving the run recoverable, and re-claims the queue's requeued active item before recording the disposition. `finalizeAgentRun` drives the forward path; `recoverFinalization` autonomously applies only the post-terminal queue disposition with no reference.
- Gate 8's in-process worker (connection sub-increment 3a): `DurableAgentRunWorker.runOneCycle` closes the queue -> runtime -> result loop on the durable stores in one recoverable order (cycle-intent ids -> claim + lease -> RunRecord ref -> intent updated -> `completeExecution` -> `finalizeAgentRun` -> disposition -> clear intent). A worker-owned single-record durable `PendingFinalizationStore` checkpoint written before the claim makes re-entry supply the same Goal/AgentRun identities so dispatcher recovery is idempotent; the reference persists before acknowledgement, closing the finalizer's deferred pre-terminal window. Recovery routes by runtime state (terminal -> `recoverFinalization`; awaiting-verification -> `finalizeAgentRun(ref)`; earlier or missing state -> re-drive with the same ids, skipping re-execution when the ref exists). Execution is an injected `AgentRunExecution` port; `AgentLoopAgentRunExecution` is its first production implementation: it runs the Gate 1-4 pipeline (governed `read-file`, persisted evidence, bounded Agent Loop, independent digest verification, application finalization) against the approved task's own source document (`sourceDocument` as target, `sourceSha256` as expected digest — no payload schema change), constructs the `ApprovedTask` directly from the WorkItem so the finalizer's taskId+sourceDocument binding holds, persists through the same `RunRecordStore` the worker's finalizer resolves from, and carries verification failure in the RunRecord (drift detection) rather than throwing. The `(targetPath, expectedSha256)` derivation is one private seam. `WorkPayload` now carries an optional caller-supplied `ExecutionInput(targetPath, expectedContentSha256)` (projected by `WorkItem.executionInput()`, persisted after `allowedTools` in both filesystem serializers with schema v1 revised in place, published through a `WorkMessagePublisher` overload as explicit caller authority — not snapshot-derived); the seam prefers the declared input and falls back to the source document, so a WorkItem executes an arbitrary governed target while the `ApprovedTask` binding stays the source document. 3b process isolation and 3c a concrete local IPC adapter are next.
- Remaining connections are dependency-ordered rather than implicitly complete: worker process isolation (3b) plus selected local IPC (3c) and the real execution port; durable controls; effect ledger/fencing; retry through additional AgentRuns; and Gate 13 typed handoff/multi-agent execution. Gate 7 owns delivery/transport, Gate 8 runtime/recovery, Gate 11 Tool execution controls, Gate 12 authenticated controls, and Gate 13 concurrency.
- Gate 8 Scheduler delivery is at-least-once rather than universally exactly-once. Stable idempotency keys, fenced leases, checkpoint/recovery, versioned state migration, orphan reclamation, and replay-safe or compensatable effects compose near-exactly-once user-visible behavior; stale fences fail closed and every effect outcome remains explicit.
- The graph projection contract types five node kinds and six endpoint-checked edge kinds (JUSTIFIED_BY, SUPERSEDES, DEPENDS_ON, MODIFIES, VERIFIED_BY, RECORDED_AS) over the Decision/Architecture/Dependency/Task/Execution domains with element provenance and derived rebuild status, keyed to one snapshot identity; `TaskImpactQuery` answers the task-to-decision-to-code-to-test chain with snapshot-traceable immutable results, and transitive DEPENDS_ON closure is deferred by decision.
- The Gate 6 run-evidence production path: `RunEvidenceGraphProducer` projects evidence-only task/artifact/execution nodes and one RECORDED_AS edge from one snapshot and one task-matched stored run record, with observation states mapped one-to-one to element freshness; it never emits edges no evidence justifies.
- Workspace RunRecord observation is bounded to the 256 most recent filesystem records; full listing and point replay remain intact, selected corrupt records stay explicitly unavailable, and no retention deletion is implied.
- The Gate 6 production graph composition: the CLI `run` path observes a 256-record recent execution window, the run's target file, safe Git index/untracked/deleted metadata, and an explicit unavailable tracked-worktree diff; it merges accepted decisions and explicit justification edges and reports bounded graph/impact counts. Inputs are preflighted before execution, and post-persist reporting cannot change the durable run exit code.
- CLI graph inputs available before execution are preflighted before evidence creation; duplicate document/target paths collapse to one target-preferred artifact. Once a finalized RunRecord is persisted, reconstructible Project Brain reporting is degradable and cannot change its durable exit code.
- Production text limits are Unicode-safe at their prefix/suffix boundary, so valid supplementary characters cannot become malformed persisted or displayed text. Governed content and artifact reads enforce byte ceilings during streaming and consume at most one overflow-detection byte after the limit; metadata size checks remain early diagnostics rather than the resource boundary.
- `GitWorkspaceCollector` holds the only external command authority, explicitly user-granted and decision-scoped: one fixed filter-free `ls-files` metadata invocation through a canonical absolute executable outside the observed project, no shell or inherited `GIT_*` overrides, project-confined discovery, watchdog timeout, and digest-only retention. Tracked worktree diff is explicitly UNAVAILABLE because verified Git comparison commands can execute required clean filters. The authority-boundary exit criterion is pinned by `WorkspaceAuthorityBoundaryIntegrationTest`; diagnostics/terminal/selection adapters remain owned by later gates.
- In-process Tool execution now has a process-wide 64-live-worker isolation ceiling: a slot remains occupied until the real daemon thread exits, and saturation fails closed with typed terminal evidence before thread creation. This prevents unbounded accumulation but cannot terminate stuck code; process isolation remains required for long-running Scheduler Tool workers. Parent-directory fsync for power-loss durability remains separately scoped architecture work.
- Runtime package dependencies are now acyclic: neutral verification decision/status/code values live in `kernel`, AgentRun finalization lives in `application`, and the enforced direction is application -> run/verification/loop/kernel, run -> loop/kernel, verification -> loop/kernel, loop -> kernel. RunRecord schema and enum names are unchanged; the project remains one Gradle module.
- The long-term OS substrate is Workspace and Project Brain -> Event/Message Bus with IPC adapters -> Agent Runtime and Scheduler -> MCP/Model Gateway -> Skill Engine and Plugin Marketplace.
- Runtime Agents communicate by typed messages rather than direct Agent-to-Agent calls once the bus boundary exists.
- Agent orchestration escalates from one worker to sequential work and only later to Producer-Reviewer, bounded fan-out/fan-in, supervisor allocation, or shallow hierarchy; one Kernel coordinator owns terminal run state.
- Every worker handoff preserves the approved task revision, common immutable Workspace snapshot, authorization, correlation, causation, budgets, and artifact or evidence references. Heartbeats and quality signals are diagnostic only.
- MCP is a core server/client interoperability layer for governed Tools, resources, Workspace views, and memory.
- Repository instructions, Tool/terminal output, model responses, MCP content, plugins, dependencies, and generated artifacts are untrusted data. Architecture owns the common provenance, secret/outbound-data, least-privilege, isolation, audit, and rollback baseline; Gates 9, 11, 12, 14, and 16 own concrete enforcement for their surfaces.
- Interfaces share one Run, approval, verification, evidence, control, recovery, and change-review API. Delivery order is API first, CLI reference surface, VS Code for in-context work, and Desktop later for supervision; the primary UX presents plan, files/diff, tests/evidence, risks, approvals, recovery, and commit readiness rather than Agent topology.
- Product milestones are V1 AI Development Experience, V2 AI Development Platform, and V3 AI Operating System; they do not replace maturity gates.
- Product milestones describe outcomes; dependency-ordered Delivery Gates may build platform foundations before every V1 interface is polished.
- Self-hosting development of Enhancer and local or hybrid model execution are separate capabilities; neither implies the other.
- Git and canonical documents remain authoritative while Project Brain supplies rebuildable Decision, Architecture, Dependency, Task, and Execution graph projections.
- Agent plugins, Skills, Tools, and workflows are separate extension types; marketplace installation never grants execution authority.
- The future Model Router defaults sensitive code to approved local models and requires policy authority for remote transmission.
- New contracts must name their current or next-gate integration consumer.
- Archon `263cf365` and meta-harness `ccab9a6` are pinned orchestration design references only. Selected patterns are restated as provider-neutral Enhancer contracts in their owning roadmap gate; their packages, prompts, file queues, Skill layouts, and provider commands are not dependencies.
- Repository documents are product inputs.
